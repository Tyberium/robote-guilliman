# Core Architecture Goals (Senior Flexes)

### 1. Hierarchical Parsing & Chunking Strategy
- Never rely on naive character splitting. Warhammer data structures require structural awareness.
- Implement a hierarchical parser that converts complex PDF layouts (unit datasheets, rules, point tables) into clean Markdown or HTML structures so the LLM retains table boundaries.
- Maintain Parent-Child relationships in metadata (e.g., matching weapon profiles or faction exceptions directly back to their parent rules/units).

### 2. Advanced Retrieval (Hybrid Search & Pre-Filtering)
- Implement Hybrid Search combining Dense Vector embeddings with Sparse BM25 keyword matching.
- Use explicit Metadata Pre-filtering (e.g., isolating query targets by `faction`, `rule_type`, or `phase`) *before* executing vector cosine similarity searches to eliminate cross-contamination between armies and slash token costs.

### 3. Data Governance & Version Control (The Errata Override Engine)
- The pipeline must handle Games Workshop's active balance updates (Balance Dataslates, Munitorum Field Manuals, FAQs).
- Implement data versioning within the vector store using `status` (e.g., `active`, `inactive`) and `superseded_by` metadata tags to handle upserts/overrides seamlessly, ensuring outdated rules are never retrieved.

### 4. Continuous Evaluation (Testing Harness)
- Budget for an automated evaluation layer (using frameworks like Ragas or TruLens) within a `tests/` directory to measure Context Precision, Faithfulness, and Answer Relevance against a hardcoded "Golden Dataset" of complex queries.

# Warhammer 40k Domain Glossary
Keep this domain logic intact when parsing and embedding text. Do not normalize or strip these terms out:

- Stratagem: Episodic game actions bound to strict phase triggers (e.g., Command Phase, Shooting Phase) and army restrictions.
